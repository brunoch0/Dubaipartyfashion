## 1. 프로젝트 개요
---
### 한 줄 정의
두바이 취향 라이프스타일 브랜드 웹사이트

### 제품 목표
두바이 커뮤니티·파티 네트워크 기반의 취향 라이프스타일 브랜드를 빠르게 검증하고, 초기 관심 고객의 대기명단(리드)을 확보한다. 이후 패션 → 파티복 → 향수 등 카테고리 확장을 수용할 수 있도록 구조를 유연하게 설계한다.

### 배경
브랜드 아이덴티티(색/로고/폰트/카피)가 아직 미확정인 상태에서 1차로 웹사이트를 구축해야 한다. 따라서 브랜드 요소는 디자인 토큰(CSS 변수)로 분리해 추후 값만 교체 가능해야 하며, 콘텐츠(글·이미지·이벤트·상품 등)는 코드 하드코딩 없이 데이터/메타데이터 기반으로 운영 가능해야 한다. 초기 범위는 랜딩/소개/콘텐츠/커뮤니티·이벤트 중심이며, 쇼핑몰은 ‘Coming Soon’ 수준의 진입점만 제공한다.

### 사용자 문제
브랜드에 관심이 생긴 사용자는 어떤 브랜드인지 빠르게 이해하고, SNS/이벤트로 이어지거나 대기명단에 가입하고 싶다. 운영자는 브랜드 톤이 확정되지 않았더라도 빠르게 사이트를 열고, 콘텐츠와 이벤트를 지속적으로 추가하면서 반응(가입/신청/클릭)을 측정하고 싶다.

### 해결 방안
랜딩·소개·블로그(SEO)·룩북/갤러리·SNS 피드·커뮤니티/이벤트·쇼핑몰(플레이스홀더)로 구성된 브랜드 웹사이트를 제공한다. 공통적으로 디자인 토큰 기반 테마, 데이터/메타데이터 기반 콘텐츠 관리, 다국어(한국어/영어/아랍어) 토글 및 RTL 고려, 모바일 우선 반응형, 분석/전환 이벤트 삽입 가능한 구조를 갖춘다.

### 차별점
브랜드 아이덴티티 미확정 상태에서도 운영 가능한 ‘디자인 토큰 우선’ 구조와, 콘텐츠 하드코딩을 배제한 ‘데이터/메타데이터 기반’ 운영을 기본값으로 둔다. 또한 한국어·영어·아랍어 3개 언어 토글과 아랍어 RTL 대응을 초기 설계에 포함해 두바이 시장 적합성을 높인다.

### 타겟 사용자
1) 잠재 고객/팔로워: 두바이 기반 또는 두바이 라이프스타일에 관심이 있는 사용자. 파티/커뮤니티/패션·향수 등 취향 기반 활동에 관심이 많고, SNS에서 유입되어 브랜드를 탐색한다. 2) 브랜드 운영자/콘텐츠 관리자: 브랜드 톤이 아직 확정되지 않았지만 빠르게 런칭이 필요하며, 블로그/룩북/이벤트를 자주 업데이트하고 대기명단·RSVP 전환을 측정하려는 팀.

### 사용자 시나리오
사용자 시나리오 A(신규 유입): 사용자가 SNS/검색으로 랜딩에 진입 → 히어로에서 브랜드 후크를 확인 → 브랜드 소개 요약을 읽고 룩북/블로그/이벤트로 탐색 → 관심이 생겨 대기명단에 이름/이메일(선택: 취향 태그)을 제출 → 확인 메시지 후 SNS/이벤트 링크로 이동.
사용자 시나리오 B(이벤트 전환): 사용자가 커뮤니티·이벤트 목록을 보고 관심 이벤트 상세로 이동 → RSVP/신청 폼 제출 → 확인 메시지 및 관련 링크 클릭.
운영자 시나리오: 운영자가 블로그 글/룩북 이미지/이벤트 정보를 메타데이터로 추가·수정 → 사이트에 반영 → 분석 이벤트로 대기명단 가입/RSVP/링크 클릭 전환을 확인.

### 핵심 지표
- 대기명단 가입 전환율(세션 대비 가입)
- 대기명단 가입 수(주/월)
- 이벤트 RSVP/신청 전환율 및 신청 수
- 핵심 링크 클릭률(SNS/이벤트/사전예약·크라우드펀딩)
- SEO 유입(검색 유입 세션, 아티클별 유입)

### 리스크
- 브랜드 아이덴티티 미확정으로 초기 메시지/비주얼 일관성이 떨어질 수 있음
- 다국어/RTL 대응 복잡도로 일정·품질 리스크 발생 가능
- 콘텐츠를 데이터로 운영하기 위한 최소한의 관리 방식(스키마/입력 규칙)이 없으면 다시 하드코딩으로 회귀할 수 있음
- 외부 SNS 임베드/플랫폼 정책 변경으로 표시가 깨질 수 있음
- 분석/전환 이벤트 설계가 늦으면 검증 목표(대기명단 확보) 학습 속도가 떨어질 수 있음

## 1. 랜딩 페이지로 브랜드 검증 및 대기명단 확보
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

랜딩 페이지에서 브랜드 후크(카피 슬롯), 브랜드 소개 요약, 대기명단 가입 폼, SNS/이벤트 링크, 사전예약/크라우드펀딩 섹션을 제공해 초기 관심 고객의 전환(대기명단 가입·링크 클릭)을 유도한다. 브랜드 요소는 디자인 토큰(CSS 변수) 기반으로 분리한다.

**수용 기준:**
```
1. ○ 히어로 영역에 브랜드 후크(카피 슬롯)와 주요 CTA(대기명단 가입 유도)가 노출된다
2. ○ 대기명단 폼은 이름, 이메일, 선택적 취향 태그 입력을 지원한다
3. ○ SNS/이벤트 링크 영역이 있으며 링크 클릭을 측정할 수 있다
4. ○ 사전예약/크라우드펀딩 섹션(현재/예정)을 노출할 수 있다
5. ○ 색/타이포/로고/카피 슬롯이 CSS 변수/디자인 토큰으로 분리되어 값 교체가 가능하다
```


### 1.1 히어로 섹션(브랜드 후크/CTA)
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

랜딩 상단 히어로에 브랜드 후크(카피 슬롯)와 대표 CTA(대기명단 가입 유도)를 노출한다. 카피/버튼 라벨은 다국어 콘텐츠 데이터로 관리하며, 스타일은 디자인 토큰(CSS 변수)로 제어한다.


#### 1.1.1 카피 슬롯/CTA 콘텐츠 모델(다국어)

히어로 영역의 텍스트/버튼은 언어별로 교체 가능한 데이터로 관리한다.
- 필수: 헤드라인, 서브카피, CTA 라벨, CTA 링크
- 선택: 배지 문구(예: New/Drop), 보조 링크 1개
- 폴백: 아랍어 미입력 시 영어, 영어 미입력 시 한국어
- 길이 가이드: 헤드라인/서브카피 최대 글자 수 권장치를 둔다
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 1.1.2 CTA 딥링크/스크롤 이동 규칙

히어로 CTA는 ‘대기명단 섹션으로 스크롤’ 또는 ‘대기명단 페이지/모달’로 연결될 수 있다.
- 링크 타입: 내부 앵커(#waitlist), 내부 경로(/waitlist), 외부 URL
- 외부 URL은 새 탭 열기 여부 옵션
- 링크 타입별 분석 이벤트 속성에 linkType을 포함
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


#### 1.1.3 히어로 비주얼(이미지/동영상) 로딩/폴백

히어로 배경 비주얼은 이미지 또는 짧은 동영상(선택)으로 운영 가능해야 한다.
- 성능: 모바일에서는 이미지 우선, 동영상은 조건부 로드(예: Wi‑Fi/저전력 모드 고려)
- 접근성: 대체 텍스트/자막(동영상) 슬롯 제공
- 폴백: 비주얼 미설정 시 브랜드 컬러 그라데이션/패턴 토큰으로 대체
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


#### 1.1.4 히어로 노출/클릭 분석 이벤트

히어로의 핵심 전환을 측정한다.
- 이벤트: hero_impression, hero_cta_click
- 속성: 언어, 페이지, 위치(hero), CTA 목적(waitlist/외부), 캠페인(UTM)
- 개인정보/보안: 개인식별정보(이메일/이름) 미수집
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


### 1.2 브랜드 소개 요약 섹션
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중

랜딩에서 브랜드 철학/가치/세계관을 짧게 요약해 신뢰를 형성하고, About/룩북/블로그로 탐색을 유도한다. 문구는 다국어 데이터로 관리한다.


### 1.3 대기명단 가입 폼
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

이름/이메일/선택적 취향 태그를 입력받아 대기명단에 등록한다. 제출 성공/실패 상태, 유효성 검사(필수/형식), 재시도 UX를 제공한다. 가입 전환 이벤트를 심을 수 있는 구조를 포함한다.


#### 1.3.1 입력 필드/유효성 검사(이름·이메일·취향 태그)

대기명단 폼은 이름(필수), 이메일(필수), 취향 태그(선택)를 제공한다.
- 이메일 형식 검증 및 중복 제출 방지(동일 이메일 재제출 시 안내)
- 필수값 누락/형식 오류 시 인라인 오류 메시지
- 제출 버튼 비활성/로딩 상태 제공
- 개인정보 안내 문구 슬롯(다국어) 포함
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 1.3.2 제출 상태 UX(성공/실패/재시도)

제출 시 성공/실패/네트워크 오류 상태를 명확히 안내한다.
- 성공: 확인 메시지와 다음 행동(예: SNS/이벤트 링크) 유도
- 실패: 오류 메시지 + 재시도 버튼
- 네트워크 오류: 재시도 및 입력값 유지
- 스팸 방지(간단한 봇 방지 수단) 적용 가능 구조
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 1.3.3 대기명단 데이터 저장/운영 최소 스키마

대기명단 제출 데이터는 코드 하드코딩 없이 저장/운영 가능해야 한다.
- 필수: 이름, 이메일, 언어, 제출 시각
- 선택: 취향 태그 배열, 유입 소스(UTM), 랜딩 위치(섹션 ID)
- 상태: 활성/차단(스팸) 플래그
- 내보내기/연동을 고려해 고유 식별자(예: submissionId) 부여
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


#### 1.3.4 대기명단 전환/분석 이벤트

대기명단 전환을 측정하기 위한 표준 이벤트를 심는다.
- 이벤트: waitlist_submit_start, waitlist_submit_success, waitlist_submit_fail
- 속성: 언어, 콘텐츠/캠페인(UTM), 위치(섹션), 에러 코드(실패 시)
- 개인정보/보안 고려: 이벤트에 이메일/이름 원문을 포함하지 않는다
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


### 1.4 SNS/이벤트 링크 섹션
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중

랜딩에서 인스타그램/스냅챗 등 SNS 및 이벤트/커뮤니티 관련 외부 링크를 노출한다. 링크 클릭을 분석 이벤트로 측정할 수 있도록 링크 메타데이터(대상/위치)를 함께 관리한다.


#### 1.4.1 링크 콘텐츠 모델(대상/위치/언어)

랜딩의 SNS/이벤트 링크를 데이터로 운영할 수 있게 링크 메타데이터 모델을 정의한다.
- 필수: label(표시명), url, linkType(sns/event/community/other), 위치(sectionId), 언어, 정렬 값
- 선택: 아이콘/플랫폼명(Instagram/Snapchat 등), 설명 문구, 새 탭 여부
- 폴백: 언어 미입력 시 기본 언어 값 사용
- 유효성: URL 형식 검증, 비공개/숨김 상태 지원
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 1.4.2 링크 노출/클릭 분석 이벤트(랜딩)

랜딩의 SNS/이벤트 링크 영역 성과를 측정한다.
- 이벤트: landing_link_impression, landing_link_click
- 속성: 언어, 위치(sectionId), linkType, 링크 대상(도메인), UTM(가능 시)
- 개인정보/보안: 개인식별정보 미수집
- 중복 방지: 동일 링크 다중 노출/클릭 시 집계 기준 정의
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 1.4.3 임베드/외부 이동 UX(새 탭/경고 문구)

외부 링크 이동 경험을 일관되게 제공한다.
- 새 탭 열기 기본 정책(예: 외부 도메인은 새 탭)
- 외부 이동 안내 문구 슬롯(다국어)
- 모바일에서 딥링크(앱 열기) 가능 시 우선 시도, 실패 시 웹으로 폴백
- 링크 깨짐(404 등) 최소 대응: 재시도/대체 링크/숨김 처리
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


### 1.5 사전예약/크라우드펀딩 섹션(현재/예정)
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중

사전예약/크라우드펀딩 정보를 현재/예정 상태로 노출하고, 외부 랜딩으로 연결한다. 노출 여부/상태/링크는 데이터로 관리하며 클릭 측정을 지원한다.


#### 1.5.1 사전예약/크라우드펀딩 상태 모델(현재/예정/숨김)

사전예약/크라우드펀딩 섹션을 상태값으로 운영한다.
- 상태: 현재(진행중), 예정(티저), 숨김
- 필수: 제목, 요약, 링크 URL, 상태, 시작/종료 일시(선택)
- 언어별 콘텐츠/링크 라벨 지원
- 폴백: 예정 상태에서는 알림/대기명단 CTA로 대체 가능
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


#### 1.5.2 전환 이벤트(사전예약 링크 클릭/티저 노출)

사전예약/크라우드펀딩 섹션의 전환을 측정한다.
- 이벤트: preorder_section_impression, preorder_link_click
- 속성: 언어, 상태(current/upcoming), 링크 대상(도메인), UTM(가능 시)
- 개인정보/보안: 개인식별정보 미수집
- A/B 또는 카피 변경 실험을 고려해 variant 속성 확장 여지
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 1.5.3 예정 상태의 대체 CTA(대기명단/알림)

진행중 링크가 없거나 예정 상태일 때도 전환이 끊기지 않도록 대체 CTA를 제공한다.
- CTA 타입: 대기명단 섹션 스크롤, 대기명단 모달/페이지, 외부 알림 채널 링크
- 카피 슬롯(다국어) + 클릭 분석 이벤트 연동
- 상태 전환(예정→현재) 시 자동으로 CTA가 링크 CTA로 교체되도록 규칙화
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중

## 3. SEO 아티클/블로그로 검색 유입 확보
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

SEO 유입을 위한 아티클/블로그 목록과 상세 페이지를 제공한다. 페이지별 메타/OG 및 구조화 데이터(Article schema)를 포함한다. 글 콘텐츠는 코드 하드코딩 없이 데이터/메타데이터 기반으로 관리한다.

**수용 기준:**
```
1. ○ 아티클 목록 페이지에서 글 목록(제목/요약/썸네일/태그/발행일)을 볼 수 있다
2. ○ 아티클 상세 페이지에서 본문, 저자, 발행일, 태그를 볼 수 있다
3. ○ 아티클 상세 페이지에 메타 태그와 OG 태그를 설정할 수 있다
4. ○ 아티클 상세 페이지에 Article 구조화 데이터를 포함한다
5. ○ 글 데이터는 하드코딩이 아닌 데이터/메타데이터로 관리된다
```


### 3.1 아티클 목록 페이지
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

아티클 목록에서 제목/요약/썸네일/태그/발행일을 카드/리스트로 노출하고, 태그/정렬 등 기본 탐색을 제공한다. 데이터/메타데이터 기반으로 운영한다.


#### 3.1.1 목록 UI 구성(카드/리스트) 및 필드 규칙

아티클 목록의 카드/리스트 UI와 표시 규칙을 정의한다.
- 필드: 제목, 요약, 썸네일, 태그, 발행일, 읽기 시간(선택)
- 표시: 말줄임/줄 수 제한, 썸네일 비율/폴백, 태그 최대 개수
- 정렬: 최신순 기본, 인기/추천(가능 시) 확장 여지
- 접근성: 키보드 포커스, 이미지 대체 텍스트
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 3.1.2 탐색(태그/검색/페이지네이션) 동작

목록 탐색 동작과 상태 유지를 정의한다.
- 태그 필터: 다중 선택 여부, 0건 처리
- 검색(선택): 제목/요약 기반, 디바운스, 결과 없음 UX
- 페이지네이션: 더보기/페이지 번호/무한 스크롤 중 1개 기본 선택(초기: 더보기)
- 상태 유지: 상세→목록 복귀 시 필터/스크롤 유지
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


#### 3.1.3 아티클 목록 데이터 모델(필수/선택)

목록/상세 공통으로 사용할 아티클 메타데이터 최소 스키마를 정의한다.
- 필수: articleId, 제목, 요약, 본문(리치 텍스트 또는 마크다운), 대표 이미지, 발행일, 언어, 상태(공개/초안)
- 선택: 태그, 저자, OG 이미지, canonical, 읽기 시간, 추천 플래그
- 폴백: 대표 이미지/요약 누락 시 기본 처리 규칙
- 검증: 날짜 형식, 상태값 제한, 언어 값 제한(ko/en/ar)
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


### 3.2 아티클 상세 페이지(본문/메타데이터)
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

아티클 상세에서 본문, 저자, 발행일, 태그를 노출한다. 본문/메타데이터는 하드코딩 없이 데이터로 관리한다.


#### 3.2.1 상세 페이지 섹션 구성(본문/메타)

아티클 상세 페이지의 섹션 구성과 표시 규칙을 정의한다.
- 섹션: 타이틀/메타(저자·발행일·태그), 커버 이미지, 본문, 관련 아티클(선택), 공유 CTA(선택)
- 본문: 마크다운/리치 텍스트 렌더링 규칙, 이미지/캡션, 인용/리스트 스타일
- 폴백: 저자/태그 없을 때 숨김, 이미지 없을 때 레이아웃 유지
- 다국어: 언어 토글에 따른 본문/메타 전환, 아랍어 RTL 고려
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 3.2.2 내비게이션(이전/다음/목록 복귀) 및 상태 유지

아티클 상세 탐색을 돕는 내비게이션을 정의한다.
- 목록 복귀: 목록의 필터/스크롤 상태 유지
- 이전/다음: 같은 태그/정렬 컨텍스트에서의 이동(가능 시)
- URL: articleId 기반 영구 링크(언어별 경로 정책 포함)
- 공유: URL 복사/공유 버튼(선택)
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


#### 3.2.3 본문 내 외부 링크 UX(새 탭/보안)

본문에 포함된 외부 링크의 UX/보안 기본 정책을 정의한다.
- 기본: 외부 도메인은 새 탭, rel="noopener noreferrer" 적용
- 링크 표시: 외부 링크 아이콘/안내(선택)
- 깨진 링크 최소 대응: 404 감지 불가 전제에서의 안내 문구/신고 링크(선택)
- 분석(선택): 본문 링크 클릭 이벤트(개인정보 미수집)
> **중요도**: 🟢 낮음 | **진행 상태**: ⚪ 작성중


### 3.3 SEO 메타/OG/구조화 데이터 설정
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

아티클 상세에 메타 태그/OG 태그를 설정하고 Article 구조화 데이터를 포함한다. 언어별(ko/en/ar) 메타데이터를 지원한다.


#### 3.3.1 페이지 메타 태그(Title/Description) 다국어

아티클 상세 페이지는 언어별 Title/Description을 출력한다.
- 메타데이터가 없을 때의 폴백 규칙(예: 제목/요약 기반 자동 생성)
- 길이 가이드(Title/Description) 준수
- canonical URL 설정(언어별 또는 기본 정책에 따름)
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 3.3.2 OG 태그(공유 카드) 설정

SNS 공유를 위한 OG 태그를 언어별로 설정한다.
- og:title, og:description, og:image, og:url, og:locale
- 이미지가 없을 때 기본 OG 이미지 폴백
- 트위터 카드 등 추가 태그 확장 여지
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


#### 3.3.3 Article 구조화 데이터(JSON-LD) 출력

Article schema 구조화 데이터를 JSON-LD로 포함한다.
- headline, datePublished, author, image, inLanguage, mainEntityOfPage
- 언어별 값 지원
- 필수 필드 누락 시 최소 안전 출력 규칙(오류 방지)
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 3.3.4 robots/noindex 정책 및 색인 제어

색인 정책을 메타데이터로 제어할 수 있다.
- 기본: 공개 아티클은 index
- 초안/비공개 상태는 noindex
- 사이트 전체 정책과 충돌 시 우선순위 정의
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


#### 3.3.5 SEO 품질 점검 체크(운영 가드레일)

운영자가 메타데이터를 빠뜨려도 품질이 무너지지 않도록 최소 체크를 둔다.
- 누락 경고 항목: title/description/OG image/발행일
- 폴백 적용 여부를 운영자가 인지할 수 있는 구조(예: 관리 화면/로그)
> **중요도**: 🟢 낮음 | **진행 상태**: ⚪ 작성중

## 4. 룩북·컬렉션 갤러리로 비주얼 탐색 제공
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중

룩북·컬렉션 갤러리에서 이미지 중심 탐색 경험을 제공한다. 이미지/항목 데이터는 메타데이터로 관리하고 코드 하드코딩을 금지한다.

**수용 기준:**
```
1. ○ 룩북/컬렉션 목록에서 갤러리 그리드로 항목을 탐색할 수 있다
2. ○ 룩북/컬렉션 상세에서 이미지(복수)와 캡션/태그 등 메타데이터를 노출할 수 있다
3. ○ 항목 데이터(이미지 URL, 제목, 태그, 정렬 등)는 데이터/메타데이터로 관리된다
4. ○ 모바일 우선 반응형 갤러리 레이아웃을 제공한다
```


### 4.1 룩북/컬렉션 목록 갤러리
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중

룩북/컬렉션 항목을 이미지 중심 그리드로 노출하고, 태그/정렬 등 기본 탐색을 제공한다. 항목 데이터는 메타데이터 기반으로 관리한다.


#### 4.1.1 갤러리 카드/그리드 UI 규칙(룩북/컬렉션)

룩북/컬렉션 목록 갤러리의 카드와 그리드 레이아웃 규칙을 정의한다.
- 레이아웃: 모바일 1열 → 태블릿 2열 → 데스크톱 3~4열(토큰 기반)
- 카드 필드: 커버 이미지, 제목, 시즌/드롭(선택), 태그(선택)
- 표시 규칙: 이미지 비율/크롭 기준, 로딩 스켈레톤, 폴백 이미지
- 접근성: 키보드 탐색, 이미지 대체 텍스트(가능 시)
- 성능: 지연 로딩(lazy load) 기본
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 4.1.2 탐색(태그/정렬) 및 상태 유지(룩북)

사용자가 원하는 스타일을 빠르게 찾도록 탐색 기능을 정의한다.
- 태그 필터: 단일 또는 다중 선택(초기: 단일), 0건 처리
- 정렬: 최신순 기본, 인기/에디터 픽(선택)
- 상태 유지: 상세→목록 복귀 시 필터/스크롤 유지, 공유 가능한 URL(쿼리)
- 빈 상태: 콘텐츠 없음/준비중 안내 문구
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


#### 4.1.3 룩북/컬렉션 메타데이터 모델(필수/선택)

룩북/컬렉션을 데이터/메타데이터로 운영하기 위한 최소 스키마를 정의한다.
- 필수: collectionId, 제목, 요약(선택 가능하나 권장), 커버 이미지, 언어, 상태(공개/초안)
- 선택: 시즌/드롭, 태그, 대표 색/무드, 정렬 우선순위, OG 이미지
- 폴백: 요약/커버 누락 시 대체 규칙
- 검증: 상태값/언어 제한(ko/en/ar), 이미지 URL 형식
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


### 4.2 룩북/컬렉션 상세(이미지/캡션/태그)
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중

룩북/컬렉션 상세에서 복수 이미지와 캡션/태그 등 메타데이터를 노출한다. 이미지 비율/로딩/모바일 레이아웃을 고려한다.


#### 4.2.1 상세 갤러리(이미지/캡션/태그) UI 규칙

룩북/컬렉션 상세에서 이미지와 캡션, 태그를 어떻게 보여줄지 정의한다.
- 구성: 히어로/커버 + 이미지 그리드(또는 캐러셀) + 캡션/크레딧(선택)
- 이미지: 확대 보기(라이트박스) 지원 여부(초기: 지원), 로딩/폴백
- 캡션: 길이 제한/줄바꿈 규칙, 언어별 캡션
- 태그: 룩/아이템 태그 표시 및 클릭 시 관련 콘텐츠로 탐색(선택)
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 4.2.2 미디어 성능/품질 가드레일(용량/포맷)

이미지 중심 페이지의 로딩 품질을 유지하기 위한 운영 기준을 정의한다.
- 권장 포맷: WebP/AVIF(가능 시), 폴백 JPEG/PNG
- 해상도: 디바이스별 srcset 기준(토큰/브레이크포인트 기반)
- 용량 가이드: 커버/상세 이미지 최대 용량 권장치
- CDN: 캐시 정책(가능 시) 및 실패 시 폴백
- 접근성: 대체 텍스트 슬롯 마련
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


#### 4.2.3 내비게이션(목록 복귀/이전·다음) 및 공유

룩북 탐색 흐름을 끊지 않도록 내비게이션과 공유를 정의한다.
- 목록 복귀: 필터/스크롤 상태 유지
- 이전·다음: 같은 정렬 컨텍스트 내 이동(가능 시)
- 공유: 컬렉션 URL 복사/공유 버튼(선택), OG 이미지 우선순위
- URL 정책: collectionId 기반 영구 링크 + 언어별 경로
> **중요도**: 🟢 낮음 | **진행 상태**: ⚪ 작성중

## 5. SNS 콘텐츠 피드 노출
> **중요도**: 🟢 낮음 | **진행 상태**: ⚪ 작성중

인스타그램/스냅챗 등 SNS 최신 콘텐츠를 임베드하거나 요약 카드 형태로 노출한다. 외부 임베드 정책 변경 리스크를 고려해 대체 표시(링크) 경로를 둔다.

**수용 기준:**
```
1. ○ SNS 피드 영역에서 최신 콘텐츠를 노출한다(임베드 또는 카드)
2. ○ 임베드가 불가하거나 실패할 경우 링크 기반 대체 UI를 제공한다
3. ○ SNS 콘텐츠 클릭을 측정할 수 있다
```


### 5.1 SNS 피드 임베드/요약 카드 노출
> **중요도**: 🟢 낮음 | **진행 상태**: ⚪ 작성중

인스타그램/스냅챗 등 SNS 최신 콘텐츠를 임베드하거나 요약 카드로 노출한다. 플랫폼별 노출 방식을 설정 가능하게 하고 데이터로 관리한다.


### 5.2 임베드 실패 시 대체 UI(링크)
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중

임베드가 불가하거나 로딩 실패 시 링크 기반 대체 UI로 전환한다. 사용자는 항상 외부 콘텐츠로 이동할 수 있어야 한다.


### 5.3 SNS 클릭/노출 측정
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중

SNS 카드/링크 클릭을 분석 이벤트로 측정한다. (가능하면 노출도 측정) 링크 대상과 위치를 함께 기록할 수 있게 한다.

## 6. 커뮤니티·이벤트 탐색 및 RSVP/신청
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

커뮤니티·이벤트 페이지에서 이벤트 목록/상세를 제공하고 RSVP/신청 폼을 통해 전환을 만든다. 이벤트 데이터는 메타데이터로 관리한다.

**수용 기준:**
```
1. ○ 이벤트 목록에서 이벤트 카드(제목/일시/장소/커버 이미지/태그)를 볼 수 있다
2. ○ 이벤트 상세에서 설명, 일시, 장소, 관련 링크를 볼 수 있다
3. ○ RSVP/신청 폼을 제출할 수 있다
4. ○ 이벤트 신청 전환 이벤트를 측정할 수 있다
5. ○ 이벤트 데이터는 하드코딩이 아닌 데이터/메타데이터로 관리된다
```


### 6.1 이벤트 목록 페이지
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

이벤트 목록에서 카드(제목/일시/장소/커버 이미지/태그)를 노출하고 탐색(태그/정렬)을 제공한다. 이벤트 데이터는 메타데이터 기반으로 관리한다.


#### 6.1.1 이벤트 카드 필드/목록 UI 규칙

이벤트 목록 카드에 어떤 정보를 어떻게 보여줄지 표준을 정의한다.
- 필드: 제목, 일시, 장소, 커버 이미지, 태그, 상태(모집중/마감)
- 표시 규칙: 긴 제목 말줄임, 이미지 비율/폴백, 태그 최대 개수
- 정렬: 최신/임박/인기(가능 시) 기본 옵션
- 접근성: 카드 전체 클릭 영역/포커스 표시
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 6.1.2 탐색(태그/정렬) 동작 및 상태 유지

태그/정렬 선택에 따른 목록 동작을 정의하고, 사용자가 탐색 맥락을 잃지 않게 한다.
- 태그 필터: 다중 선택 여부, 선택 해제 UX, 결과 0건 처리
- 정렬: 선택 값 유지(쿼리 파라미터 등) 및 공유 가능
- 스크롤: 상세→목록 돌아올 때 스크롤 위치/필터 상태 유지
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


#### 6.1.3 이벤트 데이터 모델(필수/선택)

이벤트를 데이터/메타데이터로 운영하기 위한 최소 스키마를 정의한다.
- 필수: eventId, 제목, 요약, 시작/종료 일시, 장소(텍스트), 커버 이미지, 상태(모집중/마감), 언어
- 선택: 태그, 좌석수/정원, 가격(있다면), 외부 링크, 주최자, 지도 링크
- 폴백: 이미지/요약 누락 시 기본 처리 규칙
- 검증: 일시 형식/시간대, URL 형식, 상태값 제한
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


### 6.2 이벤트 상세 페이지
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

이벤트 상세에서 설명, 일시, 장소, 관련 링크를 노출한다. 좌석/마감 여부 등 상태 값을 데이터로 관리할 수 있게 한다.


#### 6.2.1 이벤트 상세 콘텐츠 섹션 구성

이벤트 상세 페이지의 기본 섹션과 표시 규칙을 정의한다.
- 섹션: 커버/타이틀, 핵심 정보(일시/장소/태그/상태), 상세 설명, 관련 링크(지도/SNS/티켓), 주의사항
- 다국어: 언어별 콘텐츠 전환, 아랍어 RTL 레이아웃 고려
- 폴백: 특정 섹션 데이터 없을 때 숨김/대체 문구 규칙
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 6.2.2 상태값(정상/마감/정원) 표시 및 CTA 제어

이벤트 상태에 따라 CTA(신청 버튼)와 안내가 달라지도록 규칙을 정한다.
- 정상: ‘신청하기’ CTA 노출
- 마감/정원 초과: CTA 비활성 + 안내 문구 + 대체 CTA(다른 이벤트/대기명단)
- 데이터: 상태값은 메타데이터로 관리, 프론트는 규칙 기반으로 표시
- 분석: 마감 안내 노출도 측정 가능하도록 이벤트 훅 제공
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 6.2.3 일시/시간대 표시 규칙(두바이 기준)

이벤트 일시를 사용자에게 혼란 없이 보여주기 위한 규칙을 정의한다.
- 기본 시간대: Asia/Dubai(UAE)
- 표시: 날짜/요일/시간 포맷, 시작~종료 범위 표시
- 다국어: 언어별 포맷(ko/en/ar) 및 RTL에서의 숫자/구분자 처리 고려
- 캘린더 추가: iCal/구글 캘린더 링크 제공 가능하도록 데이터 슬롯 마련(선택)
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


### 6.3 RSVP/신청 폼
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

이벤트 RSVP/신청을 위한 폼을 제공하고 제출/성공/실패 상태를 안내한다. 스팸 방지(예: 간단한 봇 방지)와 개인정보 안내 문구 슬롯을 포함한다.


#### 6.3.1 RSVP 입력 필드/유효성 검사

이벤트 RSVP/신청 폼은 최소 입력만으로 전환이 가능해야 한다.
- 기본: 이름(필수), 이메일 또는 연락처(필수 중 1), 인원 수(선택), 메모(선택)
- 이벤트별 추가 질문을 메타데이터로 확장 가능(예: 드레스 코드 동의)
- 필수값/형식 오류 인라인 안내, 제출 로딩 상태 제공
- 개인정보 안내 문구 슬롯(다국어) 포함
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 6.3.2 제출 상태/마감 처리(성공/실패/정원)

이벤트 상태에 따라 제출 가능 여부와 안내가 달라야 한다.
- 정상: 제출 성공/실패/재시도 UX 제공
- 마감/정원 초과: 폼 비활성 + 안내 문구 + 대체 CTA(대기명단/다른 이벤트)
- 중복 신청(동일 이메일): 안내 후 업데이트/취소 링크 제공(가능 범위 내)
- 시간대/언어에 맞춘 일시 표시
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 6.3.3 RSVP 스팸 방지/봇 방지

자동 제출 스팸을 줄이기 위한 최소 장치를 포함한다.
- 허니팟 필드 또는 간단한 챌린지 적용 가능
- 과도한 요청(짧은 시간 다중 제출) 제한 구조
- 의심 submissions는 상태값으로 차단 처리 가능
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


#### 6.3.4 RSVP 전환/분석 이벤트

이벤트 신청 전환을 측정하기 위한 표준 이벤트를 심는다.
- 이벤트: rsvp_submit_start, rsvp_submit_success, rsvp_submit_fail
- 속성: 이벤트 ID, 언어, 유입 소스(UTM), 상태(정상/마감), 에러 코드
- 개인정보/보안 고려: 이벤트에 이메일/연락처 원문을 포함하지 않는다
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


### 6.4 이벤트 신청 전환 측정
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

이벤트 신청 제출(성공/실패), 상세→신청 시작, 관련 링크 클릭 등을 분석 이벤트로 측정할 수 있는 구조를 제공한다.


#### 6.4.1 이벤트 신청 퍼널 이벤트 정의(표준)

이벤트 상세→신청→제출까지 퍼널을 표준 이벤트로 정의한다.
- 이벤트: event_detail_view, rsvp_click, rsvp_submit_start, rsvp_submit_success, rsvp_submit_fail
- 속성: 이벤트 ID, 언어, 상태(정상/마감), 유입 소스(UTM), 에러 코드(실패)
- 개인정보/보안: 이메일/연락처 원문은 이벤트에 포함하지 않는다
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중


#### 6.4.2 링크/CTA 위치별 클릭 측정 규칙

이벤트 페이지 내 링크/CTA(신청 버튼, 지도 링크, 관련 링크)의 클릭을 위치별로 구분해 측정한다.
- 위치 값 예: detail_primary_cta, detail_secondary_link, list_card_cta
- 링크 대상(도메인/URL), 새 탭 여부 기록
- 외부 링크는 리다이렉트 페이지를 두지 않고도 측정 가능하도록 속성 설계
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


#### 6.4.3 중복/마감 상황에서의 측정 예외 처리

중복 신청(동일 이메일) 또는 마감/정원 초과 상황에서도 퍼널이 왜 끊겼는지 알 수 있어야 한다.
- rsvp_click 시 상태값 포함(정상/마감)
- rsvp_submit_fail 시 failReason(중복/마감/검증/네트워크) 기록
- 마감 상태에서 폼 비활성인 경우에도 안내 노출(‘soldout_notice_impression’) 측정 가능
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중


#### 6.4.4 대시보드용 최소 집계 지표(운영 관점)

운영자가 캠페인/언어별 성과를 빠르게 볼 수 있도록 최소 집계 지표를 정의한다.
- 지표: 상세 조회 수, 신청 클릭 수, 제출 시작 수, 제출 성공 수, 제출 실패 수
- 분해: 이벤트 ID, 언어, UTM 캠페인
- 파생: 전환율(클릭→성공, 조회→클릭)
- 개인정보: 원문 데이터 없이도 집계 가능해야 한다
> **중요도**: 🟢 낮음 | **진행 상태**: ⚪ 작성중

## 7. 향후 쇼핑몰 오픈을 위한 진입점 제공(플레이스홀더)
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중

현재는 실결제/재고/회원 기능 없이 쇼핑몰 진입점과 ‘Coming Soon’ 플레이스홀더 페이지만 제공한다. 향후 카테고리 확장을 수용할 수 있는 정보 구조를 고려한다.

**수용 기준:**
```
1. ○ 쇼핑몰 진입점(메뉴/버튼)이 존재한다
2. ○ 쇼핑몰 페이지는 ‘Coming Soon’ 플레이스홀더를 노출한다
3. ○ 현재 범위에서 실결제/재고/회원 로그인 기능은 포함하지 않는다
4. ○ 향후 카테고리 확장(패션/파티복/향수 등)을 고려한 메뉴/분류 구조의 기본 틀이 있다
```


### 7.1 쇼핑몰 진입점(메뉴/버튼)
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중

전역 내비게이션 또는 랜딩 섹션에서 쇼핑몰 진입점(메뉴/버튼)을 제공한다. 현재는 ‘Coming Soon’로 연결되며, 향후 카테고리 확장에 맞춰 메뉴 구조를 확장 가능하게 둔다.


### 7.2 쇼핑몰 플레이스홀더(Coming Soon) 페이지
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중

쇼핑몰 오픈 전 단계의 플레이스홀더 페이지를 제공한다. 현재는 실결제/재고/회원 로그인 없이, 출시 예정 안내와 관심 유도 CTA(대기명단/링크)를 노출할 수 있다.

## 8. 공통 기반: 디자인 토큰·데이터 기반 콘텐츠·다국어/RTL·반응형·분석 이벤트
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

전 페이지에 공통으로 적용되는 기반 요구사항을 정의한다: 디자인 토큰(CSS 변수) 기반 테마, 콘텐츠 데이터/메타데이터 관리(하드코딩 금지), 다국어(한국어/영어/아랍어) 토글 및 RTL 고려, 모바일 우선 반응형, 분석/전환 이벤트 삽입 구조.

**수용 기준:**
```
1. ○ 색/타이포/로고/카피 슬롯이 디자인 토큰(CSS 변수)로 분리되어 페이지 전반에 적용된다
2. ○ 아티클/룩북/이벤트 등 콘텐츠는 데이터/메타데이터로 관리되며 코드 하드코딩이 없다
3. ○ 언어 토글로 한국어/영어/아랍어를 전환할 수 있다
4. ○ 아랍어 선택 시 RTL 대응을 고려한 스타일이 적용된다
5. ○ 모바일 우선 반응형 레이아웃을 제공한다
6. ○ 대기명단 가입/이벤트 신청/핵심 링크 클릭에 대해 분석 이벤트를 심을 수 있는 구조가 있다
```


### 8.1 디자인 토큰(CSS 변수) 기반 테마
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

색/타이포/로고/카피 슬롯 등 브랜드 요소를 디자인 토큰(CSS 변수)로 분리해 전 페이지에 적용한다. 토큰 교체만으로 브랜드 아이덴티티 변경이 가능해야 한다.


### 8.2 콘텐츠 데이터/메타데이터 운영 구조
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

아티클/룩북/이벤트/링크/섹션 콘텐츠를 하드코딩 없이 데이터/메타데이터로 관리한다. 최소 입력 규칙(필수 필드/정렬/상태)을 정의할 수 있게 한다.


### 8.3 다국어 토글 및 RTL 공통 적용
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

전 페이지에서 한국어/영어/아랍어 토글을 제공하고, 아랍어 선택 시 RTL 레이아웃을 고려한 공통 스타일을 적용한다. 언어별 콘텐츠/메타데이터를 지원한다.


### 8.4 모바일 우선 반응형 레이아웃
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

전 페이지에서 모바일 우선 반응형 레이아웃을 제공한다. 그리드/타이포 스케일/터치 타깃 등 기본 원칙을 공통으로 적용한다.


### 8.5 분석 이벤트 삽입 구조
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중

대기명단 가입, 이벤트 신청, 핵심 링크 클릭 등 전환 행동에 대해 분석 이벤트를 심을 수 있는 공통 구조를 제공한다. 이벤트 이름/속성(위치, 언어, 콘텐츠 ID)을 표준화한다.
